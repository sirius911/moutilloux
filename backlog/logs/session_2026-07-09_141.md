## Problèmes d'orchestration

**Incident `ScheduleWakeup` — auto-commit prématuré avec message générique.**
Après avoir lancé l'agent `reviewer` pour `#341` en tâche de fond, j'ai
utilisé `ScheduleWakeup` à plusieurs reprises en repassant le prompt de
démarrage complet de `SESSION_ENGINE.md` (« Lis le fichier
backlog/SESSION_ENGINE.md et exécute le protocole complet ») pour patienter
jusqu'à la fin de la revue. Un de ces réveils programmés a déclenché une
ré-exécution de l'Étape 0 du protocole (résolution du parent, `git status`,
règle « working tree sale ») **avant** que je ne clôture `#341` et ne le
commite moi-même : le résultat est un commit intercalé
`sprint-37 🚧 Session précédente interrompue — état sauvegardé` contenant
exactement les fichiers de `#341` (déjà corrects sur le fond), mais avec un
message de commit générique au lieu du format attendu
`37-341 ✅ PWA minimale : ...`. Le commit n'ayant jamais été poussé vers
`origin`, il a été corrigé par un `git commit --amend` local (contenu
inchangé, message seul corrigé) avant de poursuivre. **Cause racine** :
`ScheduleWakeup` ne doit pas recevoir le prompt de (re)démarrage complet du
protocole comme `prompt` de réveil dans une session déjà en cours d'exécution
— cela invite une ré-entrée dans l'Étape 0 en parallèle du travail en cours.
**Correctif appliqué pour le reste de cette session** : abandon de
`ScheduleWakeup` pour patienter sur les agents `reviewer` — la notification
automatique de fin de tâche de fond (`task-notification`) suffit et a été
utilisée seule pour `#342`, sans incident. **Recommandation pour les
sessions futures** : ne jamais utiliser `ScheduleWakeup` avec le prompt de
démarrage de `SESSION_ENGINE.md` pendant qu'une session est déjà en cours ;
compter uniquement sur les notifications de fin de tâche de fond pour
reprendre la main après un agent asynchrone.

Par ailleurs, les deux agents `reviewer` invoqués cette session (pour `#341`
puis `#342`) ont respecté strictement leur mandat de lecture seule (vérifié
via `git status`/`gh issue view` immédiatement après chaque invocation,
avant toute autre action) — continuité du pattern déjà confirmé en session
#140, voir `feedback_reviewer_agent_overreach`.
